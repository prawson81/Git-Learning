GIT, GITBASH AND GITHUB 

 When your producing documents, code, images, any sort of files it can be important to control versions of those files and related files and any revisions made to those files. Git and GitHub are software tools that were primarily created to ensure a record is kept of things that you do whilst working on a computer, version control.

Git and GitHub allows complete records to be detailed of all work done on a project, every creation, revision, version, merge, download, save, change and more.

All the details of all the project is recorded and controlled through a repository(repo). This can be done online at GitHub or remotely using Git and a command screen(cmd) on a computer.

Git and GitHub are a systems that allows you to keep track of all history of a project and allows many people to work on the same project together, at the same time. It is mostly used by software developers. 


GIT

Git is the actual version control software, the application that you would use on your computer via a command screen(cmd), such as powershell or GitBash.

You never need to go on to GitHub to use Git software and can accomplish all things using Git that can be done online using GitHub, and vice versa.


GITHUB

GitHub is a web service that's allows you to create an account and use the Git software online, it provides an intuitive interface that makes the git processes easier and more visual.

A pull request is a proposal, to the repository originator from someone else who does not have permission to push to the origin repository or make changes on it, to pull a file with changes into the main repository. These changes/files aren't merged in to the main repository at this stage, they need to be committed to the repo by the originator to be merged.

A commit is a request to commit any changes to the repo. Essentially a save. Each commit has a unique identification code and you can/should enter a brief description of what has changed. A commit can highlight conflicts, which the repository originator must resolve prior to accepting a commit.

If you don't want to commit a change to your master branch, you can create a new branch and commit to that, so it doesn't affect the master branch or select a different existing branch to commit to, depending on your set up/requirements. This will allow you to continue working on both branches until you are happy to merge them together or keep them as they are.

The master branch is generally the main linear branch in a repo project and it shows all changes and version of the project.

You can create as many branches as you like, branches from branches and from the master branch. They don't have to go back to the master branch either. A branch could be created by each person working in collaboration on the same file in the same project, keeping their version and work separate from the master branch until the work is done, these can all be merged together into the master branch with merge conflict control using GitHub. You can view a visual representation of all branches, using the insights then network tabs in your repository page on GitHub.

A Fork is to create a copy of an entire repo in GitHub. This allows you to work on a repo without affecting the original copy. You can Fork a repo by opening a specific file in the repo and trying to modify the file. GitHub will ask you if you want to Fork the repository and edit the file, accept and your Fork copy will be created in your GitHub area.

You can also create a Fork by selecting the Fork button at the top right of the screen, whilst in the repository page you want to copy, on GitHub, then select create new fork. This button will also show you any current forks you have of that repo when you click it.


GITBASH

GitBash is a command promt(cmd) style editor that allows you to work off line on repositories, you can use this to download and upload to an online GitHub account.

Working remotely off line

note

These < > are not needed in the command code and the origin and master names are a standard naming convention used in Git/GitHub, you can use any names you like, but you must use whatever names you choose when actioning any commands that need those names.

1.    create files/code/etc, including a file named README.md, then save them in a new named folder.

2.    To convert the folder to a Git repo, open GitBash, a command screen(cmd), enter ls into the cmd to get a directory list, then cd folderlocation and        cd foldername to enter the correct directory. Use pwd to show what folder you are in and keep going until your in the folder you want. When the              cmd shows the correct folder using pwd, use git init to convert the folder to a local repo.

3.    Then use ls (LS) for a list of files, use git add to add the files in the folder to the staging area, either individually or all together. Check the files are              added using git status.

4.    Then use git commit to commit the files to the repo.

5.    Online at GitHub, create an empty repo of the same name as the folder you converted to a repo in step 2. To convert your folder to a repo, you will          have created a README.md file, so dont check this option.  

6.    Using the cmd, link the local repo to the remote repo(GitHub), with: git remote add   

7.    Then set the branch name using git branch -M  name is usually master, but can be what ever you like. If you have changed the branch name,                  remember to pwd check your cmd location and delete any obsolete branch from your repo using git branch -D.

8.    Then push your local files to the remote server(GitHub), using git push.   

** To receive updated files from the remote server (GitHub)**

9.    Pull files from the remote server, using git pull.   

(master will be what ever name you set it in step 7. & origin can be any name you use in step 6.) 


GITHUB PAGES

Allows a project to be viewed online as though its hosted on a website.

The key to this is through the repo branch, by default a branch is usually named master but you could name it anything you like, as long as the branch your working on is named gh-pages, there is automatically a secret url that you can access. 

Then by adding a html.index file using html code you can display your website.

The link to the address is in the repo: - settings - pages.

Need to remember if using commit, push and pull commands, etc, the branch name is gh-pages.


COLABIRATION

Working on the same file/project as other people and needing to include the others work into your own can create conflicts, when you and other people are working on the same thing at the same time.


Merge Conflicts

If both two or more people working on the same file change the same line(s) of code in their local file, when trying to merge their local file to the remote server(GitHub) repo this will cause a merge conflict, giving a RESOLVE CONFLICTS message.

Resolving Conflicts Using GitHub

  • On Git hub, in the repo, it will show the branch has conflicts that must be resolved and it should show the files with the conflict.

  • When you click on the resolve conflicts button it will take you into the files, showing them in a tab.

  • The conflict data should be displayed and highlighted in the first file that is automatically opened.

  • You, as the owner of the repo, then need to choose which of the two codes you want to implement and delete the other and any info supplied with the conflict, leaving only the code you want in the file.

  • You can check the whole file to make sure there are no other conflicts highlighted.

  • Once all conflicts are resolved, click the mark as resolved button.

  • Any conflicts in the other file will then be highlighted and you can resolve those, and mark them as resolved.

  • Once all conflicts are resolved, you can then click the commit merge button. This will resolve the merge conflicts.

  • This will not affect the pull request for the files to merge into the remote server(GitHub) repo.

  • You must then click the merge pull request and confirm merge buttons to complete the merge.


Resolving Conflicts remotely

It is possible to resolve conflicts on your local area, by cloning the conflict file from the other person, merging them locally then pushing them back to the remote server(GitHub) repo. - more precise instructions later!
